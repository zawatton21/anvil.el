* Vision — elisp as AI's primary language

anvil-sexp is more than a rename tool.  It is a deliberate step
toward the thesis that /elisp is the most important programming
language for AI to master/, and by extension /Emacs is the most
valuable tool an AI can operate/.

The thesis rests on properties no other language fully combines:

- *Code is data.*  Elisp round-trips through ~read~ and ~prin1~ as
  native Lisp data.  Program transformations become ordinary list
  manipulation — no CST, no AST library, no synthesis layer — and
  the output is reparseable by construction.
- *Live reflection.*  ~fboundp~, ~symbol-function~,
  ~help-function-arglist~, ~macroexpand~ and friends answer semantic
  questions about the /currently running/ program, not a static
  approximation.  The AI edits what actually exists, not what a
  parser thinks exists.
- *Macro-extensible host.*  The same language the AI edits /is/ the
  language it uses to extend the editor.  Capabilities the AI
  writes for itself compound with every session — anvil's own
  history of adding MCP tools to itself is evidence.
- *Persistent runtime.*  Unlike scripts that exit, an Emacs process
  accumulates state, telemetry, and introspection across days.
  The AI observes the downstream effect of its own edits on a
  long-running system.
- *Single substrate.*  Editing, execution, testing, version
  control, shell, filesystem, web, docs — all reachable through
  elisp calls.  Tool fragmentation (the dominant friction of
  non-Emacs AI setups) collapses into one process.

anvil-sexp's job is to close the gap between "AI can /read/ elisp"
and "AI can /refactor/ elisp the way a lisp programmer does."
Every decision in this document — reader-only backend,
preview-by-default, byte-compile + checkdoc integrated into
verify — follows from that thesis.

The complementary bet, tracked separately as anvil-ts, is tree-sitter
for non-elisp languages: a credible first-class path for the
remaining majority of the world's source code, so that nothing has
to leave the Emacs process.  Together the two modules form the
position that /Emacs + anvil is the environment in which AI editing
peaks/ — and that this shapes which tools AI users will ultimately
converge on.

* Motivation

Current state: Claude's elisp edits use ~file-replace-string~ /
~file-replace-regexp~ / ~file-batch~.  These are text-level and have
known failure modes:

- Renaming ~foo~ to ~bar~ also rewrites the substring ~foo~ inside
  docstrings, comments, and string literals.
- ~(foo 1 2)~ vs ~(foo-helper 1 2)~: regex with word-boundaries still
  mis-matches when ~foo~ appears at start or end of composite names.
- ~(defun foo ...)~ rename must also touch every call site — multiple
  ~file-batch-across~ operations and still risks false positives.
- Extracting a region into a new ~defun~ requires manual paren balancing.

Claude today mitigates this with heavy manual review ("grep first to
see candidates, then carefully review each").  That burns tool hops
and tokens on a task that is purely structural.

The fix is obvious in retrospect: parse the sexp tree, operate on the
tree, serialize.  Elisp is a lisp — this is the cheapest case in all
of programming-language tooling.

* Why reader-only for elisp

Elisp is unique among target languages because Emacs itself is the
richest possible parser and introspector for its own language.  A
generic CST parser (tree-sitter) cannot match what the native
interpreter already provides, and for elisp specifically we refuse
to carry a second backend:

1. *Reader round-trip.*  ~read-from-string~ returns live Lisp data,
   not a CST.  We can destructure, edit, and serialize with
   ~prin1-to-string~ or ~pp-to-string~.  Tree-sitter would force us
   to manipulate byte ranges and reparse after every change.
2. *Live introspection.*  ~fboundp~, ~symbol-function~,
   ~help-function-arglist~, ~function-documentation~ tell us what is
   actually loaded — semantic information tree-sitter has no access
   to.  A rename can check "is this symbol currently defined? is it
   a macro? does it have an advice list?" before proposing an edit.
3. *Macroexpansion.*  ~macroexpand-1~ / ~macroexpand-all~ reveal what
   a form actually compiles to, so we can refuse rewrites that would
   change expansion shape.
4. *Verification is trivial.*  Byte-compile, checkdoc, and sandboxed
   eval are single-call Emacs builtins — no external LSP needed.

Tree-sitter, by contrast, gives syntax only and depends on a grammar
that must be built per platform (a significant pain on Windows per
our anvil-host sentinel history).  For elisp the native path
dominates on every axis — capability, portability, performance — so
anvil-sexp is *reader-only* and will not ship any tree-sitter
fallback, ever.

Tree-sitter remains strategically important for /other/ languages
(TS, Python, Rust, Go, Ruby, …).  That work lives in a separate
module outside this document's scope — see ~Out of scope / future~.
The thesis is that Emacs + anvil becomes the most valuable AI
editing environment by combining:

- *elisp:* the richest reflective host language an AI can edit
  (this document), and
- *everything else:* tree-sitter structural queries layered on top
  of ~file-batch~ edits (future module).

* Goals

1. Operate on *Lisp data*, not byte ranges, whenever the edit is
   localized to one top-level form.  Round-trip via reader is the
   default transport.
2. Never touch strings, comments, or docstrings unless the caller
   explicitly asks (~:include 'strings~ / ~:include 'comments~).
3. Default to *preview mode* — return the proposed edit as a diff
   without writing.  Writing requires =:apply t=.  Same safety stance
   as ~code-add-field-by-map~ (Doc 06).
4. Integrate verification (byte-compile + checkdoc + optional
   sandboxed eval) so the caller can confirm a post-edit file is
   still valid in one round-trip.
5. Support project-wide rename via Doc 11's reference index — not via
   repeated Grep — in Phase 2.

* Non-goals

- Full code refactoring suite (inline let, convert-to-cl-flet, etc.).
  Start with the operations listed; others can come later if they
  prove needed.
- Multi-language (TS/Python/Ruby/Rust/Go).  This document is elisp
  only.  Non-elisp languages will be handled by a /separate/
  tree-sitter-based module tracked as future work (see below) —
  they are deliberately not mixed into anvil-sexp.
- Automatic import management.  ~file-ensure-import~ already covers
  "add require idempotently"; we don't replicate.
- Semantic type inference (cl-defun arg types, generic dispatch).

* Architecture

** Primary backend: reader + parse-sexp

The core data transport is the Lisp reader:

#+begin_src elisp
(anvil-sexp--read-file PATH)
  ;; -> list of (SEXP POSITIONS PROPS), one per top-level form.
  ;;    POSITIONS is (:start POINT :end POINT).
  ;;    PROPS carries :kind (defun/defmacro/defvar/require/...) and
  ;;    :name when derivable.
  ;;    Uses `read-from-string' in a loop over the file contents.

(anvil-sexp--write-form SEXP &key pretty)
  ;; -> string.  `prin1-to-string' for compact, `pp-to-string' for
  ;;    pretty.  Round-trip safe for well-formed elisp.
#+end_src

For locating a form by name (e.g., ~replace-defun~), we grep headers
via ~parse-sexp~ + ~syntax-ppss~ (skipping strings/comments), then
read that single sexp.  This avoids parsing the whole file when a
caller only wants one defun.

Both primitives are pure Emacs builtins, present in every supported
Emacs version, with no grammar install.

** Integrated introspection

Post-edit verification uses three builtins:

- ~byte-compile-file~ (captured warnings)
- ~checkdoc-file~
- Optional: load the file in a throwaway ~make-child-frame~-less
  process and report load errors (reuses ~anvil-offload~).

These are exposed as ~anvil-sexp-verify~ (combined) and also
individually for callers that want one signal.

** Comment / string awareness

~syntax-ppss~ gives the 4th element (string start) and 8th (comment
start) for cheap boundary checks.  The rename and replace operations
must call this before considering a match site.  Round-trip ops via
the reader inherit this for free — the reader never returns tokens
inside strings or comments.

** Edit model

All mutating operations take the same shape:

#+begin_src elisp
(plist-of-plan
 :ops
 ((:file PATH :range (BEG . END) :replacement STRING :reason STRING)
  ...)
 :summary "renamed foo → bar in 3 files, 7 sites"
 :diff-preview "..."   ; unified diff
 )
#+end_src

=:apply nil= (default) returns the plan unchanged.  =:apply t= writes
via ~file-batch-across~ (so all changes go through one audit point)
and returns the same plan with =:applied-at= timestamp added.

* Public API

** Phase 1 (Doc 11-independent)

#+begin_src elisp
(anvil-sexp-read-file PATH &key include)
  ;; Return list of forms with positions and props.  :include may
  ;; be '(strings comments) to also report string/comment ranges.
  ;; Read-only.

(anvil-sexp-surrounding-form FILE POINT &key kind)
  ;; Return the innermost top-level form containing POINT, or the
  ;; nearest of KIND (e.g. 'defun, 'let).  Read-only.

(anvil-sexp-replace-defun FILE NAME NEW-FORM &key apply)
  ;; Locate the defun/defmacro/cl-defun named NAME.  Replace its
  ;; entire form with NEW-FORM (a sexp).  NEW-FORM need not be a
  ;; defun — it can be any top-level form.  Preserves surrounding
  ;; whitespace.  Default apply=nil (preview).

(anvil-sexp-wrap-form FILE POINT WRAPPER &key apply)
  ;; Wrap the sexp at POINT with WRAPPER (a sexp containing the
  ;; placeholder symbol `|anvil-sexp-hole|`).  Example: wrap foo in
  ;; (when condition |anvil-sexp-hole|).

(anvil-sexp-macroexpand FILE NAME &key all)
  ;; Locate form by name; return macroexpansion (one level, or all if
  ;; :all t).  Read-only.

(anvil-sexp-verify FILE &key byte-compile checkdoc load)
  ;; Run byte-compile-file + checkdoc-file (and optional isolated
  ;; load via anvil-offload).  Return diagnostics as a list of
  ;; (:kind 'warning|'error :line N :message STR).
#+end_src

** Phase 2a (reader-only; no Doc 11 dependency)

Same public signatures as Phase 2b below.  The backend walks every
~.el~ file in the requested SCOPE via ~anvil-sexp--read-file~ and
scans each form's byte range for symbol-shaped matches using
~syntax-ppss~ to exclude strings and comments.  Project scope on
anvil.el (~50 files, ~50k LOC) runs in <10s on a modern machine —
fast enough to ship before Doc 11 is built.

#+begin_src elisp
(anvil-sexp-rename-symbol OLD NEW &key kinds scope apply)
  ;; Rename OLD to NEW where the occurrence is a *symbol reference*,
  ;; not a substring in a string/comment.  KINDS limits what we touch:
  ;; '(defun-name call var quote) — default all.
  ;; SCOPE: 'project / 'file / 'region / list of file paths.

(anvil-sexp-replace-call FN-NAME NEW-FORM &key scope apply)
  ;; Replace every *call* (SYM . ARGS) where SYM == FN-NAME with
  ;; NEW-FORM (which may reference positional args via %1 %2 ...).
#+end_src

** Phase 2b (uses Doc 11 — backend swap, same API)

No surface change.  Internal resolver is swapped from the Phase 2a
reader walker to Doc 11 ~anvil-defs-references~, which provides an
incrementally-updated SQLite index.  Improvements callers see:
sub-second project-scope queries, fewer false positives on custom
reader extensions, and definition-kind filtering that Phase 2a can
only approximate.

** Phase 3

#+begin_src elisp
(anvil-sexp-extract-defun FILE RANGE NEW-NAME &key arglist apply)
  ;; Wrap the given byte range in a new defun at the nearest top-level
  ;; insertion point.  ARGLIST inferred from free variables if omitted.

(anvil-sexp-inline-defun FN-NAME &key scope apply)
  ;; Replace every call site with the body of the defun, substituting
  ;; args.  Only safe for pure-expression defuns; refuse on those
  ;; with &rest, interactive forms, or self-reference.
#+end_src

** Phase 2a — cross-file rename via reader scan (no Doc 11)

Motivation: dogfood value.  anvil.el is itself an elisp codebase
and every development session that refactors it benefits
immediately from a reliable rename-symbol.  Doc 11 is the better
long-term backend, but it is a separate design and should not
gate Phase 2's delivery.

Implementation path:
- ~anvil-sexp--scan-symbol-refs FILE NAME~: for each form returned
  by ~--read-file~, text-scan the form's byte range with a
  ~\\_<NAME\\_>~ regex while consulting ~syntax-ppss~ to skip
  string literals and comments.  Return a list of
  (:kind KIND :start POS :end POS) plists where KIND classifies
  the site (defun-name / call / quote / var).
- ~anvil-sexp--project-el-files ROOT~: walk the repo with the
  existing anvil filesystem helpers, filter to ~*.el~, exclude
  ~*.elc~, ~tests/fixtures/**~, and vendored third-party trees.
- ~anvil-sexp-rename-symbol~: collect hits across the requested
  SCOPE, bucket by :kind, build a single edit plan shaped like
  Phase 1's output.  Default SCOPE is the project rooted at the
  containing git repo.  Preview by default; ~:apply t~ writes via
  ~anvil-sexp--apply-plan~.
- ~anvil-sexp-replace-call~: find every call site ~(NAME . ARGS)~
  via the same scan, render NEW-FORM with positional substitutions
  ~%1 %2 ...~, emit one edit op per site.

ERT +10: cross-file rename, string/comment exclusion, conflict
detection, kind-filtering (~:kinds '(call)~ leaves quoted symbols
alone), project-scope smoke (rename a real anvil symbol in a
scratch worktree and roll back), ~:json-false~ apply-safety (same
invariant as Phase 1), call-template positional substitution,
missing-symbol refusal, SCOPE='file and 'region variants.

*Ship criterion*: rename ~anvil-state-set~ → ~anvil-state-put~
across anvil.el yields a correct plan; ~:apply t~ writes, full
ERT suite stays green, byte-compile of every touched file
succeeds.

* Testing strategy

- Golden fixture files in ~tests/fixtures/sexp-sample/~: one per
  edit kind, hand-crafted.
- Round-trip property test: random well-formed defun → read → write
  → read → assert ~equal~.
- Integration: run Phase 1 ~replace-defun~ against anvil.el itself,
  byte-compile all files, run the full ert suite, expect 0 new
  failures.
- Verify-pipeline smoke: introduce a known warning (unused var) and
  confirm ~sexp-verify~ surfaces it.

* Platform notes

- Reader and parse-sexp are built in — no grammar install, no
  Windows-specific path concerns.
- Byte-count ranges account for CRLF vs LF: we enforce LF via
  existing .gitattributes, but internal ranges use point-based
  positions (not byte offsets) where practical.
- ~pp-to-string~ output is stable across Emacs versions we support
  (27+), but we pin formatting in tests by comparing read-back
  sexps (~equal~), not raw strings.

* Out of scope / future

- Paredit-style smartparens navigation shortcuts (interactive-mode
  concerns, not MCP).
- Macroexpansion-aware rewrites beyond the read-only
  ~sexp-macroexpand~ (e.g. rename a symbol used only as a macro
  keyword).  Would need Doc 11 Phase 4 + macro engine.
- Type-aware refactors (inference on cl-defun argument types).
- *Non-elisp languages (separate module, future work).*  A tree-sitter
  based sibling module (working name ~anvil-ts~) will provide
  language-agnostic structural locate + edit helpers for TS, Python,
  Rust, Go, Ruby, and friends, layered on top of ~file-batch~.  That
  work is intentionally a separate doc and ships independently;
  anvil-sexp must not grow a tree-sitter code path.  Strategic
  framing: elisp (this module) + anvil-ts (future) is what makes
  Emacs the most capable AI editing environment across languages.
