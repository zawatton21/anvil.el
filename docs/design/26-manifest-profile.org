* Motivation

anvil は 2026-04 時点で =anvil-bench= / =anvil-browser= / =anvil-orchestrator=
/ =anvil-worker= 等 20 モジュール、合計 *90+ MCP tool* を advertise している。
Claude Code は session 開始時に =tools/list= 全体を system prompt に詰める
ため、per-turn *manifest cost* がじわじわ効いている。

token-savior は同じ痛みに =TOKEN_SAVIOR_PROFILE= で対処している:

| Profile | Advertised | ~Tokens | Use case                         |
|---------+------------+---------+----------------------------------|
| full    |        106 |  10 950 | All capabilities                 |
| core    |         54 |   5 800 | Daily coding, no memory engine   |
| nav     |         28 |   3 100 | Read-only exploration            |
| lean    |         59 |   6 620 | Memory engine off                |
| ultra   |         17 |   2 740 | Hot tools + =ts_extended= lazy   |

=tools/list= payload を絞っても handler は生きているので、explicit call
はそのまま通る ("hidden but callable")。

anvil の per-session cost を実測してから profile を当てると、Claude の
context window が 30-50% 広く使えると期待できる。

* Goals

1. =ANVIL_PROFILE= env var + defcustom =anvil-manifest-profile= で
   tools/list の出力を subset に絞る。
2. Profile 定義はコード側 (Lisp table) ではなく data (=anvil-manifest-
   profiles.el= の =defconst=) として分離し、ユーザー定義可能に。
3. 既存の =anvil-register-tools= マクロに =:profiles= タグを追加、
   既存 registration を *壊さない* (タグ無しは =full= 扱い)。
4. =manifest-cost= MCP tool を 1 本提供 — 現在 profile の advertised
   tools 数と推定 token を返す (Claude 自身が profile 選択できる)。
5. 全 profile の token 実測値を =tests/manifest-cost-snapshot.el= に
   記録、CI で regression 検出。

* Non-goals

- 動的 (session 中の) profile 切替: JSON-RPC に profile 変更通知を
  送る仕組みは MCP 1.0 に無い。env var + 再起動で十分。
- Per-tool lazy loading: handler は常時 live を維持。advertised が
  絞られるだけ。
- User 側 (Claude Code) の =disabled_mcpServers= と重ねる: 競合する
  なら Claude Code 側を優先。
- =ultra= profile に相当する "meta-tool + tool-name lookup" (token-
  savior の =ts_extended=) は Phase 2。Phase 1 は静的 filter のみ。

** Profile 定義

(defconst anvil-manifest-profile-ultra
  '(file-read file-batch org-read-outline org-read-headline
    org-index-search bench-compare http-fetch offload-submit
    git-status git-diff-names git-log manifest-cost)
  "Hot tools only — Claude Code daily driver.")

  (append anvil-manifest-profile-ultra
          '(file-outline code-extract-pattern org-index-rebuild
            defs-search bench-last))

(defconst anvil-manifest-profile-core
  (append anvil-manifest-profile-nav
          '(file-batch-across file-replace-string file-replace-regexp

            org-update-todo-state org-add-todo org-rename-headline
            ...)))

** tools/list filter

=anvil-server-handle-tools-list= で =anvil-current-manifest-profile=
を参照、該当 profile の set に含まれるものだけ返す。profile が
=:all= (=full=) の場合 filter 無し。

#+begin_src elisp
(defcustom anvil-manifest-profile
  (intern (or (getenv "ANVIL_PROFILE") "full"))
  "Active manifest profile."
  :type '(choice (const full) (const core) (const nav)
                 (const ultra) (const lean)))

(defun anvil-current-profile-toolset ()
  (pcase anvil-manifest-profile
    ('full :all)
    ('core anvil-manifest-profile-core)
    ('nav  anvil-manifest-profile-nav)
    ('ultra anvil-manifest-profile-ultra)
    ('lean anvil-manifest-profile-lean)
    (_ (user-error "Unknown profile: %s" anvil-manifest-profile))))
#+end_src

** Manifest cost tool

#+begin_src elisp
(defun anvil-manifest-cost-handler (&optional _args)
  "Return current profile + advertised tool count + approx tokens."
  (let* ((advertised (anvil-server-advertised-tools))
         (schema-tokens (cl-reduce #'+
                          (mapcar #'anvil-tool-schema-token-count
                                  advertised)
                          :initial-value 0)))
    (list :profile anvil-manifest-profile
          :advertised-count (length advertised)
          :approx-tokens schema-tokens
          :profiles-available '(full core nav ultra lean))))
#+end_src

~anvil-tool-schema-token-count~ は tool の JSON-Schema を
=json-encode= して 1 token ≈ 4 chars で概算。厳密でない。

** Phase 1a — per-server-id profile + alias resolution (~Impl~)

Phase 0 の判定で「orchestrator worker 限定 ultra 適用」に倒したが、
単純な env 注入では効かないことが判明:

- =ANVIL_PROFILE= env は subprocess が継承するだけで、anvil-stdio.sh
  経由の emacsclient では server (main daemon) の状態に影響しない。
- Server 側 filter は *全 client 共通* のため、main session と
  orchestrator child で別 profile を返すには *server-id 別の
  filter 判定* が必要。

実装アプローチ (採用):

1. *Server-id alias 解決*: =anvil-server.el= に
   =anvil-server-id-aliases= defvar + =anvil-server--resolve-id=
   追加。virtual server-id (=emacs-eval-ultra= 等) を real id
   (=emacs-eval=) に解決して tool table を lookup。5 handler
   (initialize / tools-list / tools-call / resources-list /
   resources-templates-list / resources-read) で alias 解決を適用。
2. *Filter シグネチャ拡張*: =anvil-server-tool-filter-function= を
   =(tool-id tool-plist server-id)= に。server-id は *解決前の
   virtual id* を渡し、filter 側で profile を切り分けられるように。
3. *Per-server-id profile map*: =anvil-manifest.el= に
   =anvil-manifest-server-profiles= defcustom。既定:
   #+begin_src elisp
   '(("emacs-eval-ultra" . ultra)
     ("emacs-eval-nav"   . nav)
     ("emacs-eval-core"  . core))
   #+end_src
   =anvil-manifest-enable= で =anvil-server-id-aliases= に既定
   alias を追加 (=emacs-eval-ultra= 等を =emacs-eval= にマップ)。
4. *Back-compat*: filter signature に optional arg を追加した形なので
   legacy 2-arg 呼び出しも通る。

*Claude Code config 例* (orchestrator subprocess 用):

#+begin_src json
{
  "mcpServers": {
    "emacs-eval-ultra": {
      "type": "stdio",
      "command": "/home/<user>/.emacs.d/anvil-stdio.sh",
      "args": [
        "--server-id=emacs-eval-ultra",
        "--init-function=anvil-enable"
      ],
      "env": {}
    }
  }
}
#+end_src

Orchestrator worker (=anvil-orchestrator.el= の =spawn=) がこの
config を持つ claude-cli / codex-cli を起動すれば、child session の
=tools/list= は自動的に ultra profile に絞られる。Main session の
=emacs-eval= は変更なしで =full= 表示を維持。

テスト: =tests/anvil-manifest-test.el= に 8 ERT 追加 (alias 解決 /
per-server-id profile / enable/disable で alias install/uninstall /
end-to-end で emacs-eval-ultra が ultra 絞り込み反映)。全 24 ERT pass。

** Phase 1b — orchestrator subprocess auto-inject (~SHIPPED~ 2026-04-22)

Phase 1a までは Claude Code config 側で手動に virtual server-id
を指定する必要があった。Phase 1b で =anvil-orchestrator--spawn= が
provider の =build-cmd= に per-task MCP config を自動注入し、
ユーザーの =~/.claude.json= を触らずに child session の
=tools/list= を profile-filtered にできる。

実装範囲:
- 2 defcustom:
  - =anvil-orchestrator-manifest-profile= (default nil = opt-in。
    Phase 0 measured main-session ROI <1% につき既定無効。)
  - =anvil-orchestrator-manifest-stdio-command= (user の
    =anvil-stdio.sh= path。nil / 未存在ファイルは graceful no-op。)
- Helper 群 (claude 以外の provider でも再利用可能な stateless 函数):
  - =--manifest-config-file-path TASK-ID=: 決定論的 temp path。
    sentinel cleanup が同じファイルを見つけられるよう task-id 起因。
  - =--manifest-active-profile TASK=: opt-out (task plist の
    =:no-manifest-override t=) と defcustom を合成、返り値は
    profile symbol or nil。
  - =--manifest-stdio-ready-p=: stdio command customization 検証。
  - =--manifest-write-config TASK PROFILE=: 単一 =mcpServers= entry
    JSON を temp ファイルに書く。entry は =type=stdio=、=command=
    = customization、=args= = =[--server-id=emacs-eval-PROFILE,
    --init-function=anvil-enable]=。
  - =--manifest-inject-mcp-config CMD TASK=: pipeline stage、
    active-profile + stdio-ready なら CMD に =--mcp-config FILE=
    を append、そうでなければ CMD を素通し。
  - =--manifest-cleanup-config TASK-ID=: sentinel 側 rm。
- claude =build-cmd= は worktree flag の直後で helper を呼ぶ 1 行
  のみ追加。
- Sentinel (=--finalize=) が cleanup を task-id で呼び出し、failure
  / cancel でも temp file leak 無し。

Scope 制限: claude のみ wire-up。codex / gemini / aider は各 CLI
の MCP config flag が異なるため同 pattern で後続 phase で追加可能。
build-cmd 側の helper は provider-agnostic なので再利用前提。

Tests: 5 ERT fixture 全 pass:
- =manifest-disabled-adds-no-flag= (default nil → no flag)
- =manifest-enabled-injects-mcp-config= (flag + JSON shape 検証)
- =manifest-opt-out-suppresses-flag= (=:no-manifest-override t=)
- =manifest-missing-stdio-skips-injection= (graceful degradation)
- =manifest-config-file-path-helper= (deterministic path)

   - 編集系 tool (=file-replace-*= / =json-object-add= 等) は =core=

   - tag 付け忘れは =anvil-release-audit= の 6 本目 scanner として
     実装: 「=:read-only t= だが nav 所属でない」「新 tool id が
     どの profile にも現れない」を warning として出す。
   - ultra は author 主観 (Phase 0/1 後に measurement ベースで再
     調整、Review Q 6 参照)。
6. Q. (2026-04-20 review 追加) =ultra= 16 tools の根拠は? 客観基準
   が無いと 2 ヶ月後に誰も更新できない。
   → A. Phase 1 時点は *author 主観* (daily-driver の頻度感覚)。
   Phase 2 で =orchestrator-stats= + main session tool-call log を
   組み合わせ、*直近 30 日の tools/call frequency 上位 N* を候補
   として提示する仕組みを追加。=anvil-manifest-suggest-ultra= command
   で periodic に推薦。
   ただし頻度だけで hot 判定すると =file-outline= 等「低頻度だが
   絶対外せない」tool を落とす恐れ。frequency × "alternative cost"
   (= 同目的の他 tool 必要ツールホップ数) で重み付け。
