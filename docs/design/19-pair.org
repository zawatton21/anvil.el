~DRAFT~ (2026-04-19)

Tier 3 of Week-2 roadmap.  Extends anvil-orchestrator with a
peer-review primitive: one Claude's output is reviewed by another AI
(or a codex judge) before the user ever sees it.

* Motivation

Memory =feedback_orchestrator_role_split.md~ already establishes a
claude=architect / codex=coder pattern.  Doc 10 Phase 4b gave us the
meta-LLM judge primitive.  Doc 19 composes those into a reusable
"pair review" that:

1. Claude A produces a diff or plan.
2. Claude B / codex / gemini reviews it (critique + suggested
   amendments).
3. Judge merges the critique into an actionable punch list.

Today this composes by hand via orchestrator-submit calls.  Doc 19
packages it as one MCP call.

* Goals

1. ~pair-review DIFF :reviewers '(claude codex)~ dispatches 2+
   reviewers in parallel, collects their critiques, returns merged.
2. ~pair-iterate~ loop: apply patches → re-review → stop when judge
   says "ready" or max N rounds.
3. Reuse Doc 10 providers + concurrency caps; no new subprocess logic.

* Non-goals

- Code execution sandboxing (Doc 13 bisect does this separately).
- Human-in-the-loop prompts.  If the user wants to intervene, they
  kill the batch.

* Architecture

Thin layer over Doc 10.  ~pair-review~ constructs a DAG:

#+begin_example
  [reviewer-1]  [reviewer-2]  [reviewer-N]
        \          |            /
         \         |           /
             [judge-merge]
#+end_example

Uses ~orchestrator-submit~ with ~:depends-on~ (already supported, Doc
10 Phase 1b).  Judge prompt template is a defcustom.

* Public API

#+begin_src elisp
(anvil-pair-review DIFF &key reviewers judge system-prompt-append)
  ;; Returns batch-id.

(anvil-pair-iterate PROMPT &key reviewers judge max-rounds apply-fn)
  ;; Loop.  apply-fn called with each round's diff; caller decides
  ;; how to apply (default: show diff, no auto-apply).
#+end_src

|------------------+--------------------------------------------+--------------------|
