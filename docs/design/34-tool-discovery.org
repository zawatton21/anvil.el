* Motivation

anvil は 2026-04 時点で *90+ MCP tool* を advertise している (see Doc 26)。
Doc 26 の profile filter は tools/list の *volume* を制御するが、session 内で
Claude が「今やりたい X にどの tool が対応するか」を判断するには依然
CLAUDE.md の手書きフローチャート + memory 参照が必要で、毎 session 冒頭で
同じ探索を繰り返している。

真の問題は tool 数ではなく *intent-based discovery の不在*:

- Doc 26 manifest: *volume reduction* — session prompt の token 削減
- 本 doc (34): *semantic discovery* — 「X 用の tool はどれか」を runtime で問い合わせ可能に

両者は直交する。Doc 26 + Doc 34 が揃って初めて「advertise する分は少なく、
欲しい時は discover できる」の両立が成立する。

観測された反復探索パターン (session transcript より):
- 「org の section を読みたい」→ CLAUDE.md 再読 → ~org-read-headline~ 発見
- 「巨大ファイルから反復 block を抽出」→ 既存 tool 失念 → 結局 Read + Grep で組立て
- 「JSON 辞書に key/value 追加」→ ~json-object-add~ を思い出すまで Edit 経由
- 「同じ import を N ファイルへ」→ ~file-batch-across~ の存在を忘れて 1 つずつ

どれも tool 自体は存在し、解決パターンは CLAUDE.md に書かれている。
ただ、その *選択フローチャート* が自然言語でしか表現されていないため、
Claude が毎回 text 全体を読み直している。

* Design Principles

1. *metadata-driven, not text-driven* — tool の intent / layer は register 時に
   コード内 data として付与、runtime から query 可能に
2. *backward compatible* — 既存の ~anvil-server-register-tool{,s}~ 呼び出しは
   無変更で動く。新 key は optional
3. *loose vocabulary initially* — intent 語彙は自由 symbol で始め、Phase D
   以降で自然収束後に辞書化
4. *no tool deletion* — 本 doc の範囲では削除・統合は行わない (Phase C の
   usage データ後に別 doc で判断)
5. *avoids facade bloat* — high-level verb facade (anvil-do-edit 等) は
   痛みが見えるまで保留

* Non-goals

- tool の削除・統合 (Phase C の usage 計測後、別 doc で判断)
- CLAUDE.md の「選択フローチャート」節の即時削除 (Phase A+B 運用 2-4 週後に
  置換検討、本 doc では削除しない)
- intent 語彙の事前確定 (自然発生後に収束)
- facade verb 層 (anvil-do-*) の導入 (来期判断)

** Metadata 拡張

~anvil-server-register-tool~ の offload 系 key pass-through リストに 3 key
を追加:

#+begin_src elisp
(dolist (k '(:offload :offload-require :offload-load-path
             :offload-inherit-load-path :offload-timeout :resumable
             :intent :layer :stability))    ; ← 追加
  (when (plist-member properties k)
    (setq tool (plist-put tool k (plist-get properties k)))))
#+end_src

新 key の意味:

| Key         | 型                 | 既定値          | 例                                  |
|-------------+--------------------+-----------------+-------------------------------------|
| :intent     | list of symbol     | ~'(general)~    | ~'(file-edit batch)~                |
| :layer      | symbol             | ~'core~         | ~'core~ / ~'io~ / ~'workflow~ / ~'dev~ |
| :stability  | symbol             | ~'stable~       | ~'stable~ / ~'experimental~ / ~'deprecated~ |

plist 格納だけなので既存 tool への影響ゼロ。
未指定 tool はすべて既定値にフォールバック。

** ~anvil-tools-by-intent~ MCP tool

Signature:

#+begin_src elisp
(defun anvil-discovery-tools-by-intent-handler (query intent layer include-experimental)
  "Return MCP tools matching INTENT / LAYER / QUERY.

MCP Parameters:
  query                 - Optional regex over tool id or description (string)
  intent                - Optional intent symbol or comma-separated list
  layer                 - Optional layer: core / io / workflow / dev
  include-experimental  - If \"true\", include :stability experimental
                          (deprecated は常に除外)"
  ...)
#+end_src

Return shape (JSON):

#+begin_src js
[
  {
    "id": "file-batch",
    "layer": "core",
    "intent": ["file-edit"],
    "stability": "stable",
    "description": "Execute multiple file operations in a single call...",
    "server_id": "default"
  },
  ...
]
#+end_src

Matching 論理 (AND):
- intent 指定時: tool の ~:intent~ list と 1 つでも共通があれば hit
- layer 指定時: 完全一致
- query 指定時: ~:id~ または ~:description~ に regex match
- ~:stability deprecated~ は常に除外
- ~:stability experimental~ は ~include-experimental=true~ の時のみ含める

Sort 順:
1. layer 順 (core > io > workflow > dev)
2. intent 完全一致度 (query 指定 intent と tool intent の intersection 数)
3. id アルファベット順

Empty 結果の扱い: 空配列を返す (error にしない)。
query が regex として壊れている場合のみ ~anvil-server-tool-throw~。

** Filter 論理

~anvil-manifest--profile-toolset~ を拡張し plist-style profile を判定:

#+begin_src elisp
(defun anvil-manifest--profile-toolset (profile)
  (pcase profile
    ('full :all)
    ((or 'core 'nav 'ultra 'lean)
     ;; 既存: ID list を返す
     (symbol-value (intern (format "anvil-manifest-profile-%s" profile))))
    ((or 'agent 'edit)
     ;; 新: plist filter spec を返す
     (symbol-value (intern (format "anvil-manifest-profile-%s" profile))))

#+end_src

~anvil-manifest--visible-p~ で toolset の形状を判別し branch:
- ~:all~ → 全公開
