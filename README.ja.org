* なぜ作ったか

Anvil は先人の仕事の上に立っています。

MCP プロトコルの土台には Laurynas Biveinis 氏の [[https://github.com/laurynas-biveinis/mcp-server-lib.el][mcp-server-lib.el]] を使わせていただいています。
org 系のプリミティブは同氏の [[https://github.com/laurynas-biveinis/org-mcp][org-mcp]] を拡張・統合して組み込みました。

Anvil は、これら実績あるパッケージの上に、4 つのレイヤーを積み上げたものです:

- ファイル操作 ── テキストファイルの読み書き、大きいファイルへの局所編集
- 外部プロセス連携 ── shell コマンドや別言語 (Python など) の呼び出し
- 時間のかかる処理の対応 ── OCR、PDF 変換、ファイル一括走査など
- 上記を裏で引き受ける別 Emacs プロセス ── 走っている間も対話中の Emacs はフリーズしない

   
私は日本で 1 人電気管理事務所を営んでいます。
工場設備の保安管理、担当事業場は約 40 か所。
仕事の都合上 Windows で仕事をしていますが、長年の Emacs ユーザーでもあります。

私の仕事ではお客様の都合上、頻繁に Microsoft Office を使用しています。
おもに Excel を使って点検報告書を作成し、なにげないメモ作成、スケジュール管理には Emacs (org-mode) を使用しています。

※ノートアプリは色々渡り歩いてきましたが、最終的に Emacs へと落ちつきました。


この数ヶ月、私のワークフローは Claude を使いながら 3 つのフェーズを経て進化しました。

Anthropic のローカルファイル用エージェントであり、これによって私の仕事は劇的に効率化されました。
Excel ファイルを編集出来ることやウェブブラウザを直接操作出来ることで、私の仕事の書類整理や報告書作成、会計処理などが効率化、さらには自動化する事が出来たのです。

しかし、もっともっと効率化しようと、どんな事が出来るのか色々試していると、すぐに契約プラン (MAX プラン $200/月) の上限に達してしまう事に頭を抱えていました。

** 第 2 フェーズ ── Claude Code (CLI)

Claude Desktop アプリから Windows Terminal での Claude Code を使ったワークフローへと徐々に移行していきました。その体験は素晴しいものがありました。

shell と git との統合がずっとやりやすくなり、仕事の自動化もどんどん進んでいきましたが、Excel ベースの報告書では限界を感じていました。

ChatGPT をはじめ、Claude といった生成 AI が一番長けているのはテキストを使った作業です。

ここで、点検報告書を私が使用している Emacs (org-mode) を使って org ファイル (テキストファイル) でデータを管理するようにしたいという欲が出てきました。
※Excel ファイルはあくまで印刷用のフォーマットに特化。

これであれば、生成 AI が一番得意なファイルフォーマットで協同作業が出来るようになる。
そう考えどんどん org ファイルへの変換をしていきました。

私自身は org ファイルへとした事でデータ編集が非常にやりやすくなりました。
全てのお客様を 1 つの org ファイルで管理し、セクション毎に並びかえ。
これは org-mode を使用した事のある方であれば、皆がイメージ出来る内容だと思います。

しかし、Claude Code はどうでしょうか。

それにトークン消費も増えたような気がしました。Claude Code は大きなファイル内容を毎回読込み、書き換えた内容を全て書き戻すという処理を行っているようです。

しまいに Windows ではファイルマウントの問題から 1.2 MB 程の大きな org ファイルを編集すると、末尾の内容がどんどん消えていくではありませんか。


せっかく Claude との協同作業の為に org ベースのワークフローへと移行したのに失敗だったのか。
この便利な Emacs の機能を Claude が使えたら良いのに、、、、。

もしかしたら、Emacs ならそれが出来るのでは？


これが Anvil の開発の出発点でした。

** 第 3 フェーズ ── Anvil 経由で Emacs の中へ

そして第 3 フェーズ ── 自分で Anvil を組み上げ、Claude Code から呼べる MCP サーバーとして Emacs を立ち上げました。

これでエージェントは、走っている Emacs イメージそのものと、型付きの MCP ツールを通じて対話するようになります。
org の構造、見出し単位の編集、時間のかかる処理の振り分け ── すべてを Emacs 側の関数で処理した上で渡す。

1.2 MB の org ファイルの末尾が消える問題は、もう起こりません。
Emacs にとって大きな org ファイルは見慣れた相手で、行単位・見出し単位で正確に編集してくれます。

直近数週間で、実際に効いた例をいくつか挙げてみます。

- 1,200 行の =todo.org= 上の 29 個のサブタスクを、atomic な =file-batch= 1 コールで refile ── 1 秒未満。
- =zawatton21= → =zawatton= のリネームを、7 リポジトリ 39 ファイルにまたがって、=emacs-eval= 上の elisp ループ 1 回で完了 ── 往復 1 回。
- 点検報告書の OCR と PDF ラスタライズを =anvil-worker= にディスパッチ。スキャン中も対話中の Emacs はフリーズしません。


私が抱えていた「大きな org が編集できない」「Excel での意思疎通に限界がある」「トークン消費が膨れる」── こうした問題は、おそらく Claude をはじめとした生成 AI を使っているあなたも、どこかで似たような形で抱えているはずです。

人間が道具を作り、その道具で次を作る ── Anvil で Emacs を AI の最強ツールに鍛えていきましょう！！
Emacs ならそれが可能です。

* Anvil とは？

Anvil は Emacs の機能を [[https://modelcontextprotocol.io/][Model Context Protocol (MCP)]] ツールとして公開し、
あらゆる AI コーディングアシスタントから呼び出せるようにします。
Claude Code、Claude Desktop、GPT、Ollama/llama.cpp 経由のローカルモデルなど、
MCP を話すクライアントならどれでも対応しています。

~(anvil-enable)~ の一行で、Emacs が 50 以上のツールを備えた AI 向け作業台に変わります。
ファイル編集、org-mode 操作、git 参照、Elisp 評価など、すべて安全なサンドボックス
インターフェース越しに利用できます。

* Emacs ユーザーじゃない方へ — トークン削減目的で導入する場合

「自分は Emacs を使わないけど、Claude Code / Cursor のトークン消費を減らしたい」──
そういう動機でこのページに来た方向けの節です。

** Emacs の UI を開く必要はありません

Anvil は MCP server として裏で動くだけです。あなたが普段使っているエディタ
(VSCode / Cursor / Neovim / Claude Code 単体 / その他) を変える必要はありません。

#+begin_example
  AI クライアント ──MCP──▶ Anvil (裏の Emacs daemon) ──▶ ファイル操作
  (VSCode / Cursor /                                       (差分のみ送受信)
   Claude Code 等)
#+end_example

Emacs はバックグラウンドで daemon 1 つを常駐させるだけ。Emacs のウィンドウを
開いたり、=init.el= を書いたりする必要はありません。

** どこで効くか — 典型的な節約パターン

| シナリオ                                  | 削減率    | 何が起きているか                |
|-------------------------------------------+-----------+----------------------------------|
| 大きいファイルへの 1 行追加・削除          | *~90%+*   | フルファイル Read+Write を回避   |
| 同じファイルへの 3 件以上の編集            | *~70%*    | 1 ラウンドトリップ + atomic write |
| JSON / 設定辞書への N 件 key/value 追加   | *~73%*    | i18n、機能フラグ等               |
| TS/JS の ={ ja: "..." }= 形式データに英訳などを一括追加 | *~78%*    | マッピング JSON 1 回送信         |
| 巨大ファイルから繰り返し構造データ抽出     | *~50%*    | レガシー移行、ETL                |
| N ファイルの docstring を一括変更          | *~60-80%* | 1 コールで全ファイル分           |

数値は実測ベース ([[*効率性 — Anvil が AI トークンを節約する理由][効率性節]] にケーススタディ)。
内容次第ですが、典型的なリファクタリングセッションで *15,000〜25,000 トークン* の
削減が安定して出ています。

** 最小セットアップ (Emacs 未経験前提)

1. *Emacs を入れる* — daemon が動けば良いので普段使い設定は不要
   - Linux: =sudo apt install emacs= / =sudo dnf install emacs=
   - macOS: =brew install --cask emacs=
   - Windows: [[https://www.msys2.org/][MSYS2]] 経由 or [[https://www.gnu.org/software/emacs/download.html][gnu.org binary]]
2. *anvil を入れる* — [[*インストール][インストール]] 節の手順 (=async-installer= がもっとも手早い)
3. *daemon を起動して MCP server を上げる* — [[*クイックスタート][クイックスタート]] 節
4. *AI クライアントから接続* — [[*クライアント設定 — AI を Anvil に接続する][クライアント設定]] 節 (Claude Code / Claude Desktop のスニペットあり)
5. *CLAUDE.md (or 相当) に Anvil 優先指示を入れる* — [[*推奨スニペット（Claude Code =CLAUDE.md=）][推奨スニペット]] (任意だが効果大)

普段使う VSCode / Cursor / Neovim / Claude Code はそのままで OK。これといって Emacs Lisp の知識を使うことなく、AI のトークン消費削減だけを得られます。

** 何がコストか — 正直な話

- *初回セットアップに 15〜30 分* — Emacs インストール + anvil clone + MCP 設定
- *Emacs daemon が常駐* — 平時 50〜200 MB 程度のメモリ
- *Emacs を新しい依存として持つ* — アップデートや OS 移行時に 1 つ増える

これらと引き換えに得られるのが上記表の節約です。エージェント作業時間が
1 セッション 30 分以上、月数回以上ある人なら、初回コストはほぼ即座に
回収できます。

* 機能ハイライト

Anvil は約 50 個の MCP ツールを通じて、Emacs の機能を AI から直接呼べるようにします。
特によく効く機能を 5 つだけ挙げると:

| 機能 | 何ができるか |
|------|--------------|
| =file-batch= | 同じファイルへの複数編集を 1 往復で atomic に適用 (最大 ~90% トークン削減) |
| =org-read-outline= / =org-read-headline= | 13,000 行の org でも、見出しだけ・1 サブツリーだけを読む。全文 read 不要 |
| =anvil-worker= | 時間のかかる処理 (OCR、PDF 変換、ファイル一括走査など) を裏の別 Emacs に逃がす。対話中の Emacs はフリーズしない |
| 大規模 org ファイルの編集 | 1.2 MB 以上の org も末尾欠損なく編集 (Windows ファイルマウント問題を回避) |
| LLM クライアント非依存 | Claude Code / Claude Desktop / GPT / Ollama 経由のローカル LLM、すべて MCP で繋がる |

数値の根拠と詳細なケーススタディは [[*効率性 — Anvil が AI トークンを節約する理由]] にあります。
全モジュール一覧は [[*モジュール]] を参照。

* なぜ Anvil か？

- *LLM 非依存* — MCP 対応クライアントならどれとでも動作
- *安全設計* — 時間のかかる作業を裏の別 Emacs に逃がし、対話中の Emacs に影響を与えない
- *org-mode ネイティブ* — org の読み書き・リファクタを一級サポート
- *大容量ファイル対応* — 1.2 MB 超のファイルもデータ損失・切り捨てなしで処理
- *クロスプラットフォーム* — Linux、macOS、Windows、WSL
- *モジュラー* — 必要なものだけロード、独自モジュールで拡張可能

* 効率性 — Anvil が AI トークンを節約する理由

LLM エージェントはトークン課金です。Read/Edit のたびにバイトが飛び、
オーケストレーションのオーバーヘッドが積み重なります。Anvil のツールは
エージェントのコンテキスト予算を、書類整理ではなく推論に使えるよう設計されています。

** ファイル全体ではなく、領域に操作する

汎用ファイルツールは Read → Edit のパターンを強制し、そのたびにファイル全体を
転送します。Anvil は差分だけを送る、ピンポイントな操作を提供します。

| タスク | 汎用フロー | Anvil フロー |
|--------+------------+--------------|
| 1.2 MB の org ファイルで文字列を 1 件置換 | 1.2 MB を Read → Edit (旧文字列 + 新文字列を送信) | =anvil-file-replace-string= (パターンと置換文字列のみ) |
| 3 万行の =init.org= から見出し 1 件を読む | Read してページネーション | =anvil-org-read-headline= (サブツリーのみ) |
| 大きいファイルの末尾付近に挿入 | ファイル全体を Read → 書き直し | =anvil-file-insert-at-line= (差分のみ) |

org-mode のリファクタ操作（セクション移動、refile、分割）に限れば、
MCP 経由の操作は Read + Write 方式に比べて *トークンを 10〜20 倍節約* できます。

** 複数の編集を 1 往復にまとめる

エージェントが同じファイルに複数回編集を加えると、それぞれが独立したツール呼び出しに
なり、リクエスト・スキーマ・レスポンスのオーバーヘッドが重複します。
=anvil-file-batch= は編集ベクタを 1 回のトランザクションで適用し、
個別に発行する場合に比べて *最大 ~90% のトークン削減* を実現します。

** 重い処理は帯域外で実行

低速な操作（大規模タングル、フルツリー org スキャン、バイトコンパイル）は
ワーカーデーモンプールにディスパッチされるため、エージェントのメイン接続は
常に応答可能な状態を保ちます。

実測値: ~1 MB の =init.org= に対する =org-babel-tangle= はインプロセスで ~30 秒
かかりますが、ワーカーディスパッチなら ~2 秒で完了します。*エンドツーエンドで
~15 倍高速*、かつエージェントのメインセッションが一切ブロックされません。

** 大規模 org データベースへのスケール

テンプバッファをウォークする素朴なアプローチは、MCP リクエスト内で ~50 KB を超える
org ファイルに対してタイムアウトします。Anvil の org ツールはストリーミング正規表現
スキャナを使うため、マルチメガバイトの org ツリー（パワーユーザーが実際に運用している
モノリシックな =init.org= やジャーナルファイル）でも見出し検索・アウトライン読み出しが
高速に保たれます。

** ケーススタディ: 大規模 TypeScript リファクタリング

Anvil は理論だけの話ではなく、org-mode 専用でもありません。実際の大規模
TypeScript 移行プロジェクト — [[https://github.com/zawatton/newDTW.github.io][newDTW]] リポジトリ（デコンパイルされた HSP
アプリケーションを TypeScript に移植）— で検証されています。

- *1,400+* モジュールにわたる *16,000+* 行の TypeScript
- 実装ファイルと密結合した巨大な型定義ファイル（数千行規模）
- 同一ファイルへの多点編集が高頻度で発生

org-mode で効果を発揮した Anvil ツールが、ここでも定量的な成果をもたらしました。

*** =anvil-file-batch= — 1 ファイルへの複数同時編集

TypeScript のリファクタリングでは、1 ファイルに対してインポート追加・型定義変更・
実装置換・メソッドリネームを同時に行うことが日常的です。

実測: 130 行のソースファイルへの 6 編集。

| 方法                  | トークン | ラウンドトリップ |
|-----------------------+----------+------------------|
| 汎用 =Edit= ×6        | ~4,800   | 6                |
| =anvil-file-batch=    | ~1,500   | 1                |
| 削減率                | *~70%*   | *83%*            |

*** =anvil-file-replace-regexp= — 型・識別子の一括リネーム

=Promise<T>= → =AsyncResult<T>= へのコードベース全体の置換や、グローバル変数の
リネームを *1 コマンド* で完結できます。従来フロー（数百ファイルに grep し、個別に
read して edit）の機械的な繰り返しを一掃します。

*** =anvil-file-insert-at-line= / =-delete-lines= — 大きいファイルへの局所操作

ユースケース例: 4,000 行超の型定義ファイルへのフィールド 1 件追加、
またはグローバルステートインターフェースへの追記。

| 方法                         | トークン   |
|------------------------------+------------|
| ファイル全体を Read + Edit   | 数千       |
| =anvil-file-insert-at-line=  | 数十       |
| 削減率                       | *~90%+*    |

*** =anvil-json-object-add= — i18n / 設定辞書への一括追加

言語辞書、機能フラグ、設定マップなど、JSON オブジェクトに
多数の key/value を追記するワークフローでは、汎用の =Edit=
ツールを使うとコールごとに長いアンカー文字列を送出する必要
があります。

実測: i18n キー約 200 件を 40 件ずつ 5 バッチで追加。

| 方法                          | トークン | コール |
|-------------------------------+----------+--------|
| 汎用 =Edit= ×5                | ~30,000  | 5      |
| =anvil-json-object-add= ×5    | ~8,000   | 5      |
| 削減率                        | *~73%*   | 0      |

カンマ、閉じ波括弧、インデント検出、重複キー対応は自動。

*** =anvil-file-ensure-import= — import 行の冪等追加

多数のソースファイルに =import= 行を追加する作業は頻出
パターンですが、既に存在するかを毎回確認する手間があります。
=file-ensure-import= は対象行の存在確認、無ければ最後の
ヘッダ行の直後に 1 行挿入、の流れを 1 コールに集約します。
=:after-regex= を指定すれば Emacs の =(require ...)= 列にも
同じように使えます。

*** =anvil-file-batch-across= — 複数ファイルを 1 コールでリファクタ

N ファイルに似た変更 (docstring 更新、リネーム、import 追加)
を加えるリファクタでは、各ファイルの =operations= 配列を
まとめて送ることで全工程を 1 MCP コールで完了できます。

*** =anvil-code-extract-pattern= — 巨大ファイルからの構造データ抽出 (read-only)

レガシーコードからのデータ抽出や ETL でよくある「ファイル全体を Read
してから regex で組み立てる」を、1 MCP コールに置き換えます。
=:block-start= でブロック開始、=:block-end= でブロック終端 (=brace-balance=
は文字列内の波括弧を skip)、=:fields= 各 regex で値を捕捉、まとめて
プレーンデータで返します。ファイル本体は読み出されません。

実測: newDTW の =func492.ts= (3,407 行) から 347 件の if-block アイテム
定義を抽出。

| 方法                                | トークン     |
|-------------------------------------+--------------|
| ファイル全体 Read + 抽出スクリプト  | ~8,000 + α   |
| =anvil-code-extract-pattern= ×1     | ~3,000-5,000 |
| 削減率                              | *~50%*       |

レガシーマイグレーション、データ移行、i18n 棚卸しなど、繰り返し
構造を持つテキストを扱う作業全般で効きます。

*** =anvil-code-add-field-by-map= — TS/JS データに多言語訳文を一括追加

i18n 翻訳追加のような「同じファイルの全 ={lookup-key: "..."}= に
={add-key: "..."}= を一括で足したい」作業を、マッピング JSON を
1 回送るだけに短縮します。デフォルトは dry-run (=:apply= 既定 nil)
なので、まず preview を見てから =:apply t= で書き込む 2 段階で
安全に運用できます。

実測: newDTW の data ファイル 8 本 (各 25-100 entry) に =en= 翻訳を追加。

| 方法                                          | トークン |
|-----------------------------------------------+----------|
| 各ファイル Read + 全文置換 Write              | ~16,000  |
| =anvil-code-add-field-by-map= (マッピングのみ) | ~3,200   |
| 削減率                                        | *~78%*   |

既存 add-key の扱いは =:on-existing= で制御 (既定 =error= で fail-loud)、
誤上書きを防ぎます。

*** TypeScript が特に恩恵を受ける理由

| TypeScript の特性                         | Anvil が助ける理由                              |
|-------------------------------------------+--------------------------------------------------|
| 型定義と実装が同一ファイルに共存           | 1 ファイルへの多編集 → =file-batch= が輝く      |
| 巨大な型定義ファイル                       | 局所操作でフルファイル read を回避              |
| 頻繁なインポート追加                       | =file-insert-at-line= が定型追加に対応          |
| グローバルステートインターフェース         | フィールド追加はパターン化しやすく、スクリプト化も容易 |
| レガシー if-block / switch からの抽出     | =code-extract-pattern= が Read 全文を不要にする |
| TS/JS データに多言語訳文を一括追加          | =code-add-field-by-map= がマッピング 1 回で完結 |

newDTW の 1 リファクタリングセッションでの累積節約は *15,000〜25,000 トークン* に
のぼりました (~code-extract-pattern~ / ~code-add-field-by-map~ 投入後)。節約分は
ファイル転送ではなく、実際の推論に使われました。

** Anvil を使うべき場面

| シナリオ                                       | Anvil の効果                         |
|------------------------------------------------+--------------------------------------|
| 同一ファイルへの 3 回以上の編集               | ◎ 大幅なトークン削減                |
| 大きいファイルへの局所的な追記                | ◎ フルファイル Read を回避          |
| 型・識別子の一括リネーム                      | ◎ 1 コマンドで完結                  |
| インポート追加 / 型フィールド追記             | ○ ラウンドトリップ削減              |
| エージェント作業中の重い Emacs 操作           | ◎ ワーカーディスパッチ (~15× 高速)  |
| 新規ファイルのスクラッチ作成                  | △ 汎用の =Write= で十分             |
| ファイル・リポジトリの探索                    | × Glob / Grep の方が適切            |

** 実際のところどうなるか

大きな設定ファイルを触り、いくつかの org 見出しをリファクタし、タングルを走らせる
典型的なエージェントセッションは、本来ファイル転送オーバーヘッドに支配されます。
Anvil を使うと:

- 各編集は *変更バイトだけ* を送る。ファイル全体ではない。
- 多段階リファクタは *1 回* のバッチ呼び出しに折り畳まれる。
- 低速な Emacs 操作はエージェントのスレッドではなく *ワーカー* で実行される。

結果として、トークンあたりの有効仕事量が増えます — コンテキスト予算がコードの
再読みではなく、コードについての推論に使われます。

* クライアント設定 — AI を Anvil に接続する

Anvil が Emacs 内で起動したら、AI クライアント（Claude Desktop、Claude Code CLI、
その他 MCP 対応クライアント）に MCP サーバとして登録する必要があります。

ブリッジは小さなシェルスクリプト (=anvil-stdio.sh=) で、stdio 上で JSON-RPC を話し、
=emacsclient= 経由で動作中の Emacs デーモンに呼び出しを転送します。
マシンごとに一度インストールします:

#+begin_src
M-x anvil-server-install
#+end_src

デフォルトでは =anvil-stdio.sh= を =user-emacs-directory=（通常 =~/.emacs.d/=）に配置します。
配置先は =M-x customize-variable RET anvil-server-install-directory= で変更できます。

*前提条件:* Emacs がデーモンとして起動していること
(=server-start= または =emacs --daemon=)。Anvil は =emacsclient= 経由で接続します。

** Claude Desktop

設定ファイルの場所:

| OS                | パス                                                                                        |
|-------------------+---------------------------------------------------------------------------------------------|
| macOS             | =~/Library/Application Support/Claude/claude_desktop_config.json=                           |
| Linux             | =~/.config/Claude/claude_desktop_config.json=                                               |
| Windows (通常)    | =%APPDATA%\Claude\claude_desktop_config.json=                                               |
| Windows (UWP)     | =%LOCALAPPDATA%\Packages\Claude_<id>\LocalCache\Roaming\Claude\claude_desktop_config.json= |

Anvil は現在、内部的にツールを *2つの MCP server-id* に分けて
登録しています — =anvil=（eval / IDE 系）と =emacs-eval=（file,
org, buffer, worker など）。多くのクライアントは =mcpServers=
エントリ1つあたり1つのツールリストしか表示しないため、すべての
ツールを公開するには、同じスクリプトを別々の =--server-id= 値で
指す *2エントリ* が必要です。（将来のリリースでは server-id を
1つに統合する予定。当面は2エントリ構成がサポート設定です。）

*** macOS / Linux

#+begin_src json
{
  "mcpServers": {
    "anvil": {
      "command": "/path/to/.emacs.d/anvil-stdio.sh",
      "args": [
        "--server-id=anvil",
        "--init-function=anvil-enable",
        "--stop-function=anvil-disable"
      ]
    },
    "anvil-emacs-eval": {
      "command": "/path/to/.emacs.d/anvil-stdio.sh",
      "args": [
        "--server-id=emacs-eval"
      ]
    }
  }
}
#+end_src

2つ目のエントリは既に起動中のデーモンに接続し、file/org/buffer/
worker 系ツールを公開します。=--init-function= は1つのエントリに
だけ指定すればOK — 最初の接続で anvil が enable され、両方の
server-id にツールが登録されます。

Emacs デーモンがデフォルト以外のソケットディレクトリを使用している場合は、
*両方* のエントリの =args= に =--socket=/path/to/server= を追加してください。

*** Windows

Windows では Git Bash がシェルスクリプトの実行に必要です:

#+begin_src json
{
  "mcpServers": {
    "anvil": {
      "command": "C:\\Program Files\\Git\\bin\\bash.exe",
      "args": [
        "C:/Users/<you>/.emacs.d/anvil-stdio.sh",
        "--server-id=anvil",
        "--init-function=anvil-enable",
        "--stop-function=anvil-disable"
      ]
    },
    "anvil-emacs-eval": {
      "command": "C:\\Program Files\\Git\\bin\\bash.exe",
      "args": [
        "C:/Users/<you>/.emacs.d/anvil-stdio.sh",
        "--server-id=emacs-eval"
      ]
    }
  }
}
#+end_src

設定編集後は Claude Desktop を再起動してください。

** Claude Code (CLI)

CLI の設定は =~/.claude.json= にチャット履歴などと混在して保存されています。
*ファイル全体を上書きしないでください*。=claude mcp add= コマンドで
サーバエントリを追加します。すべてのツールを公開するには *両方* の

#+begin_src sh

  ~/.emacs.d/anvil-stdio.sh \
  --server-id=anvil \

Windows の場合は、スクリプトパスを上記の Git Bash 呼び出し形式に置き換えてください
(=command= = bash.exe、最初の =arg= = スクリプトパス)。

** オプション: ワーカープールも登録する

帯域外実行が有効なワークロード（大規模タングル、マルチ MB org スキャン、
バイトコンパイル）向けに、Anvil のワーカーモジュールはサブ Emacs デーモンを起動します。
AI クライアントから直接アドレスできるよう、別の MCP サーバとして登録することもできます:

#+begin_src json
{
  "mcpServers": {
    "anvil": { ... メインデーモン ... },
    "anvil-worker": {
      "command": "/path/to/.emacs.d/anvil-stdio.sh",

        "--socket=/path/to/.emacs.d/server/anvil-worker",

      ]

ほとんどのユーザーはこれは不要です。Anvil の =worker= モジュールがメイン接続越しに

* AI に Anvil を使わせる — CLAUDE.md の書き方
  
Anvil を接続しただけではツールが「使える状態になる」だけで、
エージェントが組み込みの =Read= / =Edit= / =Write= より Anvil を
*優先する* ようにしないと、上で説明したトークン節約は実現しません。
ほとんどの LLM クライアントはプロジェクト単位の指示ファイル
（Claude Code なら =CLAUDE.md=、Cursor なら =.cursorrules= 等）
を読むので、以下のスニペットを貼り付けてください。

** 推奨スニペット（Claude Code =CLAUDE.md=）

#+begin_src markdown
## ファイル編集

該当する場合は、組み込みの Read/Edit/Write より Anvil MCP ツールを
優先する。差分のみを送り、複数編集を 1 ラウンドトリップに集約し、
フルファイル read を回避できる。

- `anvil-file-batch` — 同一ファイルへの 3 つ以上の編集（1 コールに集約）
- `anvil-file-replace-string` / `anvil-file-replace-regexp` —
  ピンポイント置換。ファイル全体を read 不要
- `anvil-file-insert-at-line` / `anvil-file-delete-lines` /
  `anvil-file-append` — 行単位の局所操作

組み込みの `Edit` は小さな単発編集のみに使う。同一ファイルに 3 箇所
以上編集するなら必ず `anvil-file-batch` を使う。

## org-mode

セクション移動・refile・分割、または大きい org からの単一見出し読み取りは、
Read+Write ではなく `anvil-org-*` ツールを使う。トークン消費が 10〜20 倍
少ない。

- `anvil-org-read-headline` — 単一サブツリーの読み取り
- `anvil-org-read-outline` — body なしのアウトライン表示
- `anvil-org-edit-body` / `anvil-org-rename-headline` /
  `anvil-org-update-todo-state` — 局所的な org 編集

## 重い処理 — ワーカーディスパッチ

長時間の Emacs 操作（大規模タングル、バイトコンパイル、マルチ MB
org スキャン、フルツリー検索）はメインデーモンで実行してはいけない。
他のすべてのツール呼び出しがブロックされる。ワーカープール経由で
ディスパッチすること。

- Anvil 内から呼ぶ elisp: ~1 秒を超えそうなものは raw `eval` ではなく
  `anvil-worker-call` を優先する。
- ワーカーを別 MCP サーバとして登録している場合（README 「オプション:
  ワーカープールも登録する」参照）、重い `eval` は
  `mcp__anvil-worker__eval` を直接ターゲットにする。メインセッションの
  応答性が保たれる。

ワーカー使用すべきだったサイン: メイン MCP セッションが数秒間ツール
呼び出しを受け付けなくなった。

## スケジュールタスク（cron）

`anvil-cron` タスクが設定されている場合（lint、ヘルスチェック、バッチ
インデクサ等）、その作業を場当たり的に再実装しない。cron MCP ツール経由で
状態確認・トリガーする:

- `anvil-cron-list` — 登録タスクとスケジュール一覧
- `anvil-cron-status` — 最終実行時刻、状態、最近の失敗
- `anvil-cron-run` — 登録済みタスクの即時実行

新規アドホックスクリプトを書く前に `anvil-cron-list` を確認する。
すでにジョブとして定義されているかもしれない。

** なぜこれが必要か

明示的な指示がないと、エージェントはどの MCP サーバが接続されていても
最大公約数的なツール（=Read=、=Edit=、=Write=）にフォールバックします。
README 冒頭のトークン節約表は、エージェントが実際に Anvil ツールを
*選ぶ* ことを前提にしています。その判断を駆動するのは指示ファイルで
あって、Anvil 自身ではありません。

* FAQ — 他のパッケージとの違いは？

** …claude-code-ide との違い?

両者は異なる問題を解決していて、併用できます。
=claude-code-ide= は
Claude Code のインタラクティブ UI を eat 経由で Emacs の /中に/
持ち込みます。MCP サーフェスは主に IDE 寄り (xref / diagnostics /
buffer metadata)。ファイル編集は Claude Code 組み込みの Read / Edit /
Write をファイルシステム直で走らせます。

anvil はその逆です: スタンドアロンの MCP サーバとして /Emacs 自体/
— file 操作、org primitives、worker dispatch、elisp introspection —
を任意の MCP クライアントに公開します。実務では私は両方動かしていて、
claude-code-ide で Emacs 内インライン Claude セッション、anvil は
別 MCP サーバとして claude-code-ide /と/ スタンドアロンの ~claude~ CLI
両方から呼ばれます。

** …「任意 elisp eval」型 MCP との違い?

単発 eval 型 MCP (例: [[https://github.com/steveyegge/efrit][efrit]]) は哲学的に純粋です: クライアント側ロジックゼロ、
エージェントが全タスクで elisp を書きます。トレードオフは /機能ごとの承認切り分け/:
単一 eval ツールを =:read-only= として登録できないので、読み取りを
自動承認し書き込みだけ確認を要求、という分け方ができません。

anvil は約 50 の named primitive を公開し、該当するものは =:read-only=
hint 付きで登録されています。Claude Code の =/permissions= と綺麗に噛み合い、
~org-read-outline~ / ~file-read~ / ~elisp-describe-function~ は自動承認、
~file-batch~ / ~org-edit-body~ / ~file-replace-string~ は確認要求、という
分離が可能です。anvil も ~emacs-eval~ / ~emacs-eval-async~ を escape hatch
として備えていますが、それが 80% のパスではない、という位置づけです。

** …bash から =emacsclient= を呼ぶ方式との違い?

私自身そこから始めました。離れた理由:

- *機能ごとに承認を切り分けられない。* 全 =emacsclient -e '...'= は =/permissions= からは
  "Bash" として一括で見えます。MCP なら tool 単位で =:read-only= hint
  を登録でき、読みは自動承認・書きはプロンプト、の分離ができます。
- *スキーマがない。* エージェントは prompt から elisp API を覚える
  必要があります。MCP は tool list を system schema として公開するので、
  毎回思い出させなくても正しいツールが選ばれます。
- *エスケープ地獄。* bash を通した複数行 elisp は極めて脆く、
  日本語パスや regex を含むと特に悲惨です。
- *構造化された返り値。* MCP tool は typed JSON を返します。
  =emacsclient= は印字された sexp を自力で parse することになります。

中身は同じ Emacs。MCP 層はそこに *型情報と機能ごとの承認* を持たせた入り口、というだけです。

** …org-mcp との違い?

[[https://github.com/laurynas-biveinis/org-mcp][Org-mcp]] は MCP 上の org 編集 primitives に絞って強い package です。
anvil はその primitives (拡張 + bug fix 込み) を同梱しつつ、
file / subprocess / heavy-op / worker-pool レイヤを足しているので、
単一の MCP サーバで org に限らない作業フロー全体をカバーします。
org-mcp 単体で満足している場合、anvil は置換品ではなく /org-mcp の
発想を file/subprocess/worker まで広げたもの/ とお考えください。

** …Emacs を一度も触ったことが無いんだけど?

問題ありません。Anvil はバックグラウンド daemon として動く MCP server
で、あなたが Emacs の UI を開く必要は一切ありません。普段使っている
エディタ (VSCode / Cursor / Neovim / Claude Code 単体 等) はそのまま
で OK です。

実際に必要なのは:

- Emacs を 1 度インストールして =emacs --version= で確認
- =--fg-daemon= で常駐させる (起動スクリプトはリポジトリに同梱)
- AI クライアントの MCP 設定に 1 ブロック追加

詳しい手順は [[*Emacs ユーザーじゃない方へ — トークン削減目的で導入する場合][Emacs ユーザーじゃない方へ]] 節にまとめてあります。
Emacs Lisp の知識は不要、AI のトークン消費削減だけを得られます。
